这节讲的是 **虚拟内存（Virtual Memory）总体概念**，可以按“是什么—能做什么—怎么实现”来理解。

------

## 1. 地址空间（Address Space）

- **线性地址空间**：就是一串从 0 开始、连续递增的非负整数地址：${0,1,2,\dots}$。
- **虚拟地址空间**：进程看到的地址集合，通常大小为 $N=2^n$，就是 ${0,\dots,N-1}$。
- **物理地址空间**：真实 DRAM 的地址集合，大小为 $M=2^m$，${0,\dots,M-1}$。

关键：**每个进程都有自己独立的虚拟地址空间**，看起来像独享一大块内存，实际上被映射到同一块物理内存里，互相隔离。

------

## 2. 为什么需要虚拟内存（VM 的三大作用）

1. **作为缓存：用 DRAM 给磁盘做 cache**
   - 从概念上看：虚拟内存就是一个存放在磁盘上的大数组，按页（page）来分块。
   - DRAM 是其中一小部分的缓存：
     - 虚拟页（VP）：虚拟地址空间中的页。
     - 物理页（PP）：物理内存中的页。
   - **页大小**：通常 $P=2^p$ 字节，常见 4KB 或更大。
2. **作为内存管理工具**
   - 每个进程看到的都是 **统一的线性地址布局**：代码段、数据段、堆、共享库、栈等位置固定。
   - 系统可以自由选择“虚拟页 $\rightarrow$ 物理页”的映射，便于：
     - 动态内存分配和回收（malloc/free, brk 等）。
     - 不同进程共享同一份代码/数据（比如共享库）。
3. **作为保护机制**
   - 在页表项（PTE）里加上 **权限位**：读/写/执行/特权等。
   - MMU 在每次访问时检查权限：
     - 不允许的访问（比如用户态写只读页、访问内核页）会触发异常。
   - 不同进程的地址空间互相隔离，用户态也不能随便动内核。

------

## 3. VM 作为缓存：页表、缺页与工作集

### 3.1 页表（Page Table）

- **页表是一个数组**，元素是页表项 PTE。
- 每个 PTE 至少包含：
  - 有效位（Valid bit）：该虚拟页现在是否在物理内存里。
  - 物理页号 PPN（或者在磁盘上的地址）。
  - 权限位（读写执行等）。

每个进程都有自己的页表，存在内核的 DRAM 里。

### 3.2 Page Hit / Page Fault

- **Page hit**：访问的虚拟页已经在物理内存中（PTE 有效且映射了某个 PPN）。
- **Page fault**：访问的页不在内存，MMU 发现 PTE 无效，触发缺页异常：
  1. 硬件触发异常，跳到内核的页错误处理程序。
  2. 内核选择一个受害页（victim）要被换出，如有脏位则先写回磁盘。
  3. 从磁盘把需要的虚拟页读入某个物理页，更新页表项为有效。
  4. 返回用户态，**重新执行那条指令**（这次就 hit 了）。
- 这种在第一次访问时才加载到内存的策略叫 **按需调页（demand paging）**。

### 3.3 局部性与工作集

- 程序在一段时间内访问的活跃虚拟页集合叫 **工作集（working set）**。
- 若“单个进程工作集大小 < 物理内存大小”，在初始 compulsory miss 后性能不错。
- 若“**所有进程工作集之和 > 物理内存**”，系统会频繁换页、来回在磁盘和内存之间拷贝，几乎没时间干正事，这叫 **thrashing（抖动）**。

------

## 4. VM 作为内存管理工具

### 4.1 每个进程都有统一布局

典型 64 位 Linux 进程虚拟地址空间大致是：

- 低地址：代码段（.text）、只读数据（.rodata）
- 可写数据（.data, .bss）
- 堆（heap，`brk` / `malloc` 扩展）
- 共享库的内存映射区域（memory-mapped region for shared libraries）
- 用户栈（stack，从高地址向下生长）
- 最高处是内核的虚拟地址空间（对用户程序不可见）

### 4.2 简化链接与装载（Linking & Loading）

- **链接**：
  - 编译器/链接器可以默认让每个程序的代码、数据、堆起始地址固定（虚拟地址），不用考虑物理位置。
- **装载（execve）**：
  - 内核为 `.text`、`.data` 段分配虚拟页，**先建立 PTE 但标记为无效**。
  - 真正的代码/数据在第一次访问时，由虚拟内存系统从可执行文件里按页拷贝进来。
  - 这样可以减少启动时一次性加载的开销，也便于按需加载和共享。

### 4.3 共享代码与数据

- 不同进程可以把自己的某些虚拟页映射到同一物理页，比如：
  - 只读的共享库代码。
  - 共享内存段。
- 这样既节省内存，又实现进程间通信。

------

## 5. VM 作为保护机制

- 在每个 PTE 中增加：
  - Read / Write / Execute 权限位。
  - Supervised（内核专用）位等。
- MMU 在地址转换时检查：
  - 当前 CPU 模式（用户/内核）是否允许访问这个页。
  - 操作类型是否符合权限。
- 违规访问会触发异常（如段错误 segfault），保证：
  - 进程之间不会互相破坏内存。
  - 用户态不能随意访问/修改内核。

------

## 6. 地址转换：页表 + TLB + 多级页表

### 6.1 把虚拟地址拆开看

设虚拟地址空间大小为 $N=2^n$，物理地址空间 $M=2^m$，页大小 $P=2^p$ 字节。

- 虚拟地址 VA 拆成：
  - VPN：虚拟页号（高 $n-p$ 位）
  - VPO：虚拟页内偏移（低 $p$ 位）
- 物理地址 PA 拆成：
  - PPN：物理页号（高 $m-p$ 位）
  - PPO：物理页内偏移（低 $p$ 位，和 VPO 相同）

地址转换 MAP 的含义：

- 若 $MAP(a)=a'$：说明虚拟地址 $a$ 的数据在物理地址 $a'$。
- 若 $MAP(a)=\emptyset$：说明该虚拟地址不在物理内存（要么无效，要么在磁盘上）。

### 6.2 用页表做地址转换

基本过程（不考虑 cache/TLB）：

1. CPU 发出虚拟地址 VA。
2. MMU 用页表基址寄存器（PTBR）和 VPN 找到该虚拟页对应的 PTE。
3. 若 PTE 有效：
   - 取出 PPN，和 VPO 拼成物理地址 PA。
4. 用 PA 访问 cache / 内存，取到数据。

若 PTE 无效，则触发 page fault，走前面说的缺页处理流程。

### 6.3 TLB：加速地址转换

- 页表在内存里，**每次转换都访问 DRAM 会很慢**。
- 所以在 MMU 里放一个小的、set-associative 的硬件 cache，称为：
  - **TLB（Translation Lookaside Buffer）**。
- TLB 里缓存的是最近用的少数几项 PTE：
  - key：VPN（拆为 TLB tag + index）
  - value：PTE（里面有 PPN、权限等）

访问流程（常见情况）：

1. 给出 VA，MMU 用 VPN 索引 TLB。
2. **TLB hit**：直接拿到 PPN，不用再访问内存中的页表。
3. 拼出 PA，去 L1 cache / 内存。

**TLB miss** 时，MMU 需要再访问一次内存里的页表得到 PTE，然后填入 TLB，所以多了一次内存访问。
 因为局部性，TLB 命中率通常很高，所以平均开销很小。

### 6.4 多级页表（Multi-level Page Tables）

问题：如果地址空间很大，单级页表会巨大无比（如 48 位地址空间、4KB 页、8 字节 PTE，单表就要几百 GB）。

解决：**多级页表**，比如两级：

- 1 级页表：每个 PTE 指向一个 2 级页表页（或为 null）。
- 2 级页表：才真正存放具体虚拟页的 PTE。
- 只有用到的那部分地址空间才需要对应的 2 级页表页，未使用的大片虚拟地址区域只在 1 级表里有一项 null，不需要 2 级表页。

推广到 $k$ 级页表时，虚拟地址被拆成：

$$
 \text{VPN}_1, \text{VPN}_2, \dots, \text{VPN}_k, \text{VPO}
 $$

逐级索引，一级一级地找到下一层页表，最终得到 PPN，再和 VPO 拼成物理地址。

------

## 7. 整体小结（从程序员 & 系统视角）

- **程序员视角**：
  - 觉得自己有一大片连续的、私有的虚拟内存。
  - 不用担心和别的进程撞地址、也不用关心真实物理内存布局。
  - 访问非法地址就 segfault。
- **系统视角**：
  - 把 DRAM 当作磁盘上虚拟内存空间的 cache。
  - 借助局部性，使得大多数访问都能在物理内存内完成，只有少量缺页需要磁盘。
  - 通过灵活的映射，简化内存管理和装载，支持共享与保护。
  - 通过页表、TLB、多级页表等机制，在性能和空间开销之间折中。
