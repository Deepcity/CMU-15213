# 15 Exceptional Control Flow: Signals

***MOSTLY LEARNED***

## 核心脉络

- 异常控制流（ECF）贯穿系统多个层次：异常、进程上下文切换、信号、以及应用级的非局部跳转（setjmp/longjmp）。
- 本讲聚焦三块：Shell 背景、Signals、非局部跳转。

## 一、信号（Signals）

**1）是什么**
 信号是内核发送给进程的小消息，用来通知某种事件发生；每种信号有默认动作（终止、终止并转储、停止直到 SIGCONT、忽略）。进程收到信号时，内核依据“待决(pending)∧未屏蔽(~blocked)”计算可递送集合并逐个处理。

**2）安装处理器（handler）**
 通过 `signal`/`sigaction` 为某个信号绑定处理函数；`SIG_IGN` 表示忽略，`SIG_DFL` 表示恢复默认动作；`sigaction` 版本更可移植，并可设置 `SA_RESTART` 等标志。  

**3）阻塞/解阻塞与待决**
 对某些信号可临时屏蔽（`sigprocmask`），屏蔽期间到达的该类信号变为“待决”；解除屏蔽后再递送。内核还会**隐式**屏蔽正在处理的同类信号，防止同类 handler 重入。

**4）编写“安全”的 handler**

- 尽量简短，只做“设标志”这类工作；
- 只调用 **异步信号安全** 的函数（如 `write/_exit/wait/kill`），避免 `printf/malloc/exit` 等；
- 共享数据的读写区间可用临时阻塞保护；
- 全局标志用 `volatile sig_atomic_t`。

**5）与子进程回收的竞态**
 等待子进程退出常见竞态：如果先 `fork`，子进程很快退出而父进程尚未把它加入“作业表”，`SIGCHLD` 可能在不恰当的时机到达。解决思路：

- 在关键区（例如向作业表“加入/删除”）前先阻塞 `SIGCHLD`，关键区后再恢复；
- 等待信号推荐用 `sigsuspend` 实现**原子的“解除阻塞并睡眠”**，避免“先解除阻塞就错过信号”的竞态。

## 二、Shell 背景（为什么需要信号）

Shell 运行前台/后台作业，用户通过键盘（如 Ctrl-C/Ctrl-Z）对**前台进程组**整体发送信号；为正确回收后台子进程、避免“僵尸进程”和内核资源泄漏，Shell 必须正确处理 `SIGCHLD`。

## 三、非局部跳转（setjmp/longjmp）

**1）用途与机制**
 `setjmp` 记录当前寄存器/栈指针/PC 等上下文；`longjmp` 恢复该上下文并让 `setjmp`“第二次返回”，返回值变为 `longjmp` 的参数。常用于**错误恢复**和与信号处理结合的“快速栈展开”。  

**2）限制**
 必须遵守调用栈纪律：只能跳回**仍在栈上、尚未返回**的调用环境。

**3）示例感知**
 讲义给出“深层函数检测错误后直接跳回调用者”的示例，主调方用 `switch(setjmp(...))` 区分不同错误码。

## 速记小结

- **Signals = 进程级异常处理**：定义 handler + 合理屏蔽/解屏蔽 + 使用异步信号安全函数。
- **避免竞态**：关键区阻塞 `SIGCHLD`，等待用 `sigsuspend`。
- **非局部跳转**：`setjmp` 标记、`longjmp` 返回；适合错误恢复但要遵循栈纪律。
