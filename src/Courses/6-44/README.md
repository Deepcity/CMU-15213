大致判断如下（以 **stride = 1** 行最清晰）：

- **L1D ≈ 32 KB**
   在 16k 与 32k 行带宽最高（≈31 GB/s），到 **64k** 行明显下降到 ≈28 GB/s，说明工作集从完全命中 L1 转为主要命中下一级。→ L1 容量约在 **32 KB** 附近。
- **L2 ≈ 512 KB**
   在 **64k～512k** 行，带宽基本维持在 ≈28 GB/s 的高平台；到 **1024k（1 MB）** 时出现第一次明显台阶（≈26 GB/s），结合 512k 仍保持高平台，推断 L2 容量约 **512 KB**（1 MB 开始越界时出现降幅）。
- **LLC(L3) ≈ 16～32 MB（≈24 MB）**
   从 **2 MB～16 MB** 仍维持较高带宽（≈23～24 GB/s），到 **32 MB** 出现显著断崖（≈15 GB/s），再到 **64 MB/128 MB** 进一步跌到 ≈11/10 GB/s 的内存带宽区间。→ LLC 容量落在 **16–32 MB** 之间，粗估 **≈24 MB**。

**为什么这样判定：**
 “Memory Mountain” 的容量边界通常表现为 **随工作集大小增加** 在带宽曲面上的“台阶/断崖”。

- L1→L2：32k→64k 的明显降幅；
- L2→LLC：512k→1 MB 开始降；
- LLC→内存：16 MB 仍高，**32 MB** 处明显断崖。

**注意**：具体数值会受预取器、缓存包含/独享策略、行大小、超线程/频率、测量步长（没有 24 MB 这一行）等影响；因此 LLC 只能给出 **区间估计**。如果你愿意，我可以把脚本再改一下，在 16–32 MB 之间做更细的工作集扫描（例如 20/24/28 MB）来更精确定位 L3 容量拐点。

除了以上分析，还可以注意到：

- **L1D 与 L2 容量总和 ≈ 32 KB + 512 KB = 544 KB**
   这与实验结果中 L1D 与 L2 容量总和在 32 KB 到 512 KB 之间的分布一致。
- **LLC 容量 ≈ 16 MB**
   这与实验结果中 LLC 容量在 16 MB 到 32 MB 之间的分布一致。

在同大小的块读写，不同的 stride 时，带宽会有差异。这显然是由于 stride 不同，导致缓存未命中的次数不同，从而影响带宽。局部性原理的作用。
